# Feature Specification: 旅行计划创建 (Travel Plan Create)

**Feature Branch**: `001-travel-plan-create`

**Created**: 2026-06-02

**Status**: Draft

**Input**: User description: 智能旅行助手 - 旅行计划创建模块，帮助用户通过填写目的地、日期、交通方式及偏好信息，自动生成定制化的旅行规划方案。详见 `prototype-requirements.md`。

## Clarifications

### Session 2026-06-02

- Q: 旅行计划生成后的交互方式（可编辑页面/只读页面/PDF文档） → A: 不在本次功能范围内，先不考虑生成后的展示与编辑

## Out of Scope

- 旅行计划生成后的结果展示、在线编辑、重新生成等交互功能（留待后续迭代）
- 旅行计划的分享、导出、收藏等功能
- 旅行计划的实时预订、支付等交易功能
- 历史旅行计划的管理（查看、对比、复制旧计划）

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 创建基础旅行计划 (Priority: P1)

用户输入目的地城市、选择出行起止日期、选择交通方式后，提交生成一份基础旅行计划。这是最核心的用户旅程，覆盖最小可用的旅行规划功能。

**Why this priority**: 这是产品的核心价值主张——用户只需提供最少的必要信息（目的地、日期、交通方式），即可获得一份可用的旅行计划。没有这个流程，产品无法交付任何价值。

**Independent Test**: 用户仅填写目的地、起止日期和交通方式，不填写住宿偏好和旅行偏好，即可成功提交并生成一份包含每日行程安排的基础旅行计划。

**Acceptance Scenarios**:

1. **Given** 用户在旅行计划创建表单页面, **When** 用户输入目的地城市（如"北京"）、选择出发日期和返回日期、选择交通方式（如"高铁"），所有必填项验证通过, **Then** 用户点击"生成旅行计划"按钮，系统成功生成一份包含每日行程安排的基础旅行计划
2. **Given** 用户未填写目的地城市, **When** 用户点击"生成旅行计划", **Then** 系统提示"目的地为必填项"，阻止提交
3. **Given** 用户未选择出发日期或返回日期, **When** 用户点击"生成旅行计划", **Then** 系统提示"出行日期为必填项"，阻止提交

---

### User Story 2 - 使用偏好定制旅行计划 (Priority: P2)

用户在基础信息之上，进一步选择住宿偏好、多个旅行偏好标签（如美食、自然风光、历史文化等），生成高度个性化的旅行计划。

**Why this priority**: 这是产品的差异化竞争力——根据用户的个性化偏好生成定制行程。相比 P1 的基础计划，此流程能显著提升用户满意度和旅行体验。

**Independent Test**: 用户填写目的地、日期、交通方式，并选择住宿偏好（如"高档型酒店"）和多个旅行偏好标签（如"美食"、"历史文化"），系统生成的旅行计划中推荐的活动和路线与所选偏好高度匹配。

**Acceptance Scenarios**:

1. **Given** 用户已填写目的地、日期和交通方式, **When** 用户选择住宿偏好为"高档型酒店"，并选择旅行偏好标签"美食"和"历史文化", **Then** 系统生成的旅行计划中包含当地特色餐厅推荐和历史文化景点
2. **Given** 用户选择了 4 个旅行偏好标签, **When** 系统生成旅行计划, **Then** 每日行程合理覆盖用户选择的多个偏好类型，而非仅关注单一偏好

---

### User Story 3 - 添加特殊服务要求 (Priority: P3)

用户在旅行计划创建表单中填写额外的特殊服务要求或备注信息（如"带老人出行，行程不宜太紧凑"、"需要无障碍设施"等），系统在生成旅行计划时考虑这些特殊需求。

**Why this priority**: 特殊服务要求是可选的补充信息，用于覆盖更细分的用户场景，提升特定人群的旅行体验。不填写此字段不影响核心功能的使用。

**Independent Test**: 用户在表单的"特殊服务要求"字段中填写文字说明后提交，系统生成的旅行计划中体现出对用户特殊需求的响应（如行程节奏调整、无障碍设施推荐等）。

**Acceptance Scenarios**:

1. **Given** 用户填写了特殊服务要求"带小孩出行，需要亲子友好景点", **When** 系统生成旅行计划, **Then** 行程中包含适合儿童的景点和活动推荐
2. **Given** 用户未填写特殊服务要求, **When** 用户提交生成旅行计划, **Then** 系统正常生成旅行计划，不受该可选字段空值的影响

---

### Edge Cases

- **目的地城市无法识别**：用户输入了不存在的城市名、错别字或非中国大陆城市（如海外城市），系统应提示"未找到该目的地"并引导用户从城市列表中选择
- **出发日期晚于返回日期**：用户选择的出发日期比返回日期晚，系统应提示日期不合法并阻止提交
- **出发日期为过去日期**：用户选择的出发日期早于当前日期，系统应提示"出行日期不能早于今天"并阻止提交
- **出行天数超出合理范围**：用户选择了超过 30 天的出行天数，系统应提示"出行天数上限为 30 天"或给出合理的警告
- **网络异常导致生成失败**：用户在提交后，大模型调用超时或失败，系统应提示"生成失败，请稍后重试"并提供重试入口
- **城市搜索无结果**：用户在目的地城市搜索框中输入内容，但自动补全无匹配结果，应显示"未找到匹配城市"的提示

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供旅行计划创建表单页面，引导用户填写目的地、日期、交通方式、住宿偏好、旅行偏好和特殊服务要求
- **FR-002**: 目的地城市为必填项，系统必须提供城市搜索与自动补全功能，用户输入关键字后展示匹配的城市列表供选择
- **FR-003**: 出行开始日期和结束日期为必填项，系统必须在用户选择日期后自动计算并显示出行天数
- **FR-004**: 用户可直接调整出行天数，系统必须同步更新结束日期以保持数据一致性
- **FR-005**: 用户必须从预定义列表中选择出行交通方式（飞机、高铁、自驾、大巴等）
- **FR-006**: 用户必须从预定义列表中选择住宿偏好（经济型酒店、舒适型酒店、高档型酒店、豪华酒店、民宿等）
- **FR-007**: 用户可选择多个旅行偏好标签（景点观光、美食、自然风光、历史文化、购物体验、探险、文化体验、休闲娱乐等），选择数量无硬性上限
- **FR-008**: 特殊服务要求为可填项，用户可输入自由文本描述额外需求
- **FR-009**: 所有必填项（目的地、开始日期、结束日期、交通方式、住宿偏好）验证通过后，用户方可提交旅行计划生成请求
- **FR-010**: 系统必须基于用户输入的全部信息，利用大语言模型生成完整的个性化旅行规划方案
- **FR-011**: 系统必须在生成过程中向用户展示加载状态提示
- **FR-012**: 如果旅行计划生成失败，系统必须展示错误信息并提供重试选项

### Key Entities

- **旅行计划 (Travel Plan)**: 代表用户的一次完整旅行规划，包含目的地城市、出行起止日期、出行天数、交通方式、住宿偏好、旅行偏好标签、特殊服务要求，以及生成的行程结果
- **目的地城市 (Destination City)**: 代表用户计划前往的城市，具有城市名称和地理位置信息，用于城市搜索与自动补全
- **偏好标签 (Preference Tag)**: 代表用户感兴趣的旅行体验类型（如美食、自然风光、历史文化等），用户可选择多个标签组合
- **每日行程 (Daily Itinerary)**: 旅行计划的组成部分，代表某一天的具体行程安排，包含景点推荐、餐饮推荐、活动安排等

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户能在 2 分钟内完成旅行计划表单填写并成功提交（从进入页面到点击提交按钮）
- **SC-002**: 90% 的用户首次填写表单时无需查看帮助文档即可理解各字段含义并完成填写
- **SC-003**: 旅行计划生成成功率达到 95% 以上（不因系统错误导致生成失败）
- **SC-004**: 旅行计划生成响应时间在 30 秒以内，加载状态反馈延迟不超过 2 秒
- **SC-005**: 城市搜索自动补全在用户输入 2 个字符后返回匹配结果，响应延迟不超过 1 秒

## Assumptions

- **A-001**: 用户群体为中国大陆用户，出行目的地主要为中国大陆城市，城市搜索数据源覆盖中国地级市及以上城市
- **A-002**: 系统已具备可用的大语言模型 API 访问能力（参考项目环境变量 `LLM_API_KEY`）
- **A-003**: 表单页面为响应式 Web 页面，同时支持桌面端和移动端浏览器访问
- **A-004**: 交通方式、住宿偏好和旅行偏好标签为预定义的枚举值，由产品配置维护，不在用户端动态增删
- **A-005**: 用户无需注册或登录即可使用旅行计划创建功能，无需用户身份验证
- **A-006**: 生成的旅行计划为静态文本内容，不包含实时预订、支付等交易功能（此功能为后续迭代范围）
- **A-007**: 用户拥有稳定的互联网连接以支持表单提交和计划生成
