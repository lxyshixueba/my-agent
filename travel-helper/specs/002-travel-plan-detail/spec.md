# Feature Specification: 旅行计划详情查看与编辑 (Travel Plan Detail)

**Feature Branch**: `002-travel-plan-detail`

**Created**: 2026-06-05

**Status**: Draft

**Input**: User description: "用户在001模块 在首页表单中填写目的地城市、旅行日期、偏好、预算、交通及住宿类型等信息。点击"开始规划"按钮后，系统会显示加载进度条，并很快生成结果页面，随后加载成功，该页面会清晰展示行程概览、预算明细、景点地图、每日行程详情和天气信息 如果用户需要个性化调整，可以点击"编辑行程"按钮，自由调整景点顺序或删除某个景点。 规划完成后，通过"导出行程"下拉菜单，即可将最终方案轻松保存为图片或 PDF 文件，方便随时查阅。 参考原型图"

## Clarifications

### Session 2026-06-05

- Q: 编辑行程的范围（仅调整顺序和删除景点，还是可修改景点详情、住宿、餐饮等） → A: 支持调整景点顺序和删除景点，以及编辑景点详情（如游玩时间、景点选择等）
- Q: 导出格式优先级（图片和 PDF 是否都需要，哪种为主） → A: 图片和 PDF 都支持，用户可以按需选择
- Q: 天气信息的展示粒度（逐日天气还是概览天气） → A: 在每日行程详情页展示当天的天气信息

### Session 2026-06-05 (Speckit Clarify)

- Q: 数据持久化与用户识别策略（用户如何再次访问已创建的旅行计划） → A: 仅使用浏览器 localStorage 存储，用户只能在同一设备同一浏览器上查看之前创建的旅行计划，关闭或清除浏览器数据后计划将丢失
- Q: 编辑行程的交互方式（调整景点顺序和删除的具体交互方式） → A: 拖拽排序（用户直接拖动景点卡片调整顺序）+ 每个景点卡片上的删除按钮
- Q: 天气信息的数据来源与更新策略（实时查询还是生成时缓存） → A: 行程生成时一次性获取并缓存天气数据，用户查看时直接展示
- Q: 高德地图 MCP 服务的具体使用方式（API Key 配置、服务部署位置、使用的 API 范围） → A: 使用高德地图 MCP 服务（API Key 通过环境变量 `AMAP_WEB_KEY` 配置），MCP 服务部署在后端作为代理层，调用高德 POI 搜索、天气查询、路线规划等 API，返回 JSON 格式数据

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 查看旅行计划概览 (Priority: P1)

用户完成旅行计划创建后，系统自动生成结果页面。用户在概览页可以清晰看到：旅行计划基本信息（目的地、日期范围、计划简介）、预算明细（按景点门票、酒店住宿、餐饮交通、餐饮美食分类展示）、目的地地图（标注景点和住宿位置）、以及每日行程概览列表。这是用户验证旅行计划是否满足需求的第一站。

**Why this priority**: 这是旅行计划创建后的首要环节——用户需要直观、全面地了解系统生成的规划方案是否合理。没有概览能力，用户无法判断计划质量，也就无法进行后续的个性化调整或导出。

**Independent Test**: 用户完成旅行计划创建后，可以直接看到概览页面，包含预算明细（按分类展示各项费用和总费用）、目的地地图（带景点和住宿标记）、以及每日行程列表（点击可跳转）。

**Acceptance Scenarios**:

1. **Given** 用户刚完成旅行计划创建, **When** 系统生成完成并跳转至概览页, **Then** 页面展示计划基本信息（目的地、日期范围、计划简介）、预算明细（分类预算和总费用）、目的地地图、每日行程概览列表
2. **Given** 用户在概览页查看预算明细, **When** 页面加载完成, **Then** 预算明细按景点门票、酒店住宿、餐饮交通、餐饮美食四个分类展示各项金额和预估总费用
3. **Given** 用户在概览页查看地图, **When** 地图加载完成, **Then** 地图上标注了行程中涉及的景点位置和住宿位置
4. **Given** 用户在概览页查看每日行程列表, **When** 用户点击某一天（如"第1天"）, **Then** 页面跳转至该天的行程详情页

---

### User Story 2 - 查看逐日行程详情 (Priority: P2)

用户在概览页点击某一天后，进入该天的行程详情页。详情页按板块展示：日程安排（时间段与活动对应）、景点详情（带图片、游玩时间、景点特色、推荐信息）、住宿安排（酒店名称、房型、地址、入住退房时间）、餐饮安排（早/午/晚餐推荐）、交通安排（航班、火车等信息）、以及当日天气。

**Why this priority**: 逐日详情是用户深入了解行程的核心入口。用户需要在每天的时间线中验证景点选择是否合理、住宿餐饮是否满意，这是决定是否需要对行程进行调整的关键环节。

**Independent Test**: 用户从概览页进入某一天的详情页后，可以看到当天所有行程安排信息，包括景点详情卡片（含图片、游玩时间、景点特色）、住宿信息、餐饮推荐、交通方案和天气信息。

**Acceptance Scenarios**:

1. **Given** 用户在概览页点击"第1天", **When** 页面跳转至第1天详情, **Then** 页面展示当日的日程时间线、景点详情、住宿安排、餐饮安排、交通安排和天气信息
2. **Given** 用户查看景点详情卡片, **When** 卡片加载完成, **Then** 展示景点图片、名称、游玩时间、景点特色、推荐信息和详细描述
3. **Given** 用户查看住宿安排, **When** 信息加载完成, **Then** 展示酒店名称、房型、地址、入住时间和退房时间
4. **Given** 用户查看餐饮安排, **When** 信息加载完成, **Then** 展示早餐、午餐、晚餐的推荐方案
5. **Given** 用户查看交通安排, **When** 信息加载完成, **Then** 展示当日涉及的航班或火车等交通信息
6. **Given** 用户查看天气信息, **When** 信息加载完成, **Then** 展示当日目的地的天气情况

---

### User Story 3 - 编辑行程调整景点 (Priority: P3)

用户对生成的行程不满意时，可以编辑某一天的行程，自由调整景点顺序或删除某个景点。

**Why this priority**: 个性化调整是提升用户满意度的重要功能。AI 生成的行程可能不完全符合用户期望，用户需要有灵活的编辑能力来优化行程。

**Independent Test**: 用户进入某天的详情页后，点击编辑入口，可以在编辑界面中调整景点顺序或删除某个景点，保存后更新生效。

**Acceptance Scenarios**:

1. **Given** 用户正在查看某一天的行程详情, **When** 用户点击"编辑行程"按钮, **Then** 页面打开编辑界面，展示当日所有景点、住宿、餐饮、交通的可编辑内容
2. **Given** 用户在编辑界面中, **When** 用户通过拖拽操作调整景点卡片顺序, **Then** 景点排列顺序实时更新，保存后生效
3. **Given** 用户在编辑界面中, **When** 用户点击某个景点卡片上的删除按钮, **Then** 系统提示确认后该景点从当日行程中移除，保存后生效
4. **Given** 用户完成编辑操作, **When** 用户点击"保存"按钮, **Then** 编辑后的行程更新并展示在详情页中

---

### User Story 4 - 导出旅行计划 (Priority: P4)

用户完成行程查看和编辑后，可以通过导出功能将最终方案保存为图片或 PDF 文件。

**Why this priority**: 导出功能让用户可以将旅行方案随身携带，方便在旅行中随时查阅。这是提升用户旅行体验的最后一公里。

**Independent Test**: 用户在概览页点击"导出行程"后，可以选择图片或 PDF 格式，系统将旅行计划导出为对应格式的文件供用户下载保存。

**Acceptance Scenarios**:

1. **Given** 用户正在查看旅行计划概览, **When** 用户点击"导出行程"按钮, **Then** 系统提供导出格式选项（图片/PDF）
2. **Given** 用户选择导出格式后, **When** 导出操作完成, **Then** 系统生成对应格式的文件并提供下载

---

### User Story 5 - 重新规划行程 (Priority: P5)

用户对当前行程不满意时，可以选择重新规划整个行程，系统确认后重新生成。

**Why this priority**: 重新规划是兜底方案——当用户对生成的行程整体不满意时，可以通过重新规划获得新的方案。使用频率相对较低，但不可或缺。

**Independent Test**: 用户在概览页点击"重新规划"按钮，系统弹出确认提示，用户确认后系统重新生成旅行计划。

**Acceptance Scenarios**:

1. **Given** 用户正在查看旅行计划概览, **When** 用户点击"重新规划"按钮, **Then** 系统弹出确认弹窗，提示"重新规划将覆盖当前的行程安排"
2. **Given** 用户在确认弹窗中点击"确认", **When** 确认操作完成, **Then** 系统重新生成旅行计划并更新页面内容
3. **Given** 用户在确认弹窗中点击"取消", **When** 取消操作完成, **Then** 弹窗关闭，当前行程保持不变

---

### Edge Cases

- **行程生成失败**：大模型 API 调用失败或超时，概览页展示错误信息并提供重试选项
- **导出时网络不稳定**：导出过程中网络连接中断，系统应提示"导出失败"并提供重试
- **编辑后保存失败**：编辑行程时保存接口调用失败，系统应保留用户的编辑内容并提示重试
- **行程天数过多**：用户创建了超过合理天数（如 30 天以上）的行程，逐日详情页应按天分组展示，并提供快速导航
- **景点信息缺失**：某个景点缺少图片或详细描述，系统应展示占位图或"暂无描述"提示
- **地图加载失败**：目的地地图加载超时或失败，概览页地图区域展示占位图和重试按钮
- **用户快速切换天数**：用户频繁切换不同天的行程详情，系统应正确加载对应天数的数据
- **重新规划时数据丢失**：用户编辑过的行程被重新规划覆盖，系统应在确认弹窗中明确提示用户编辑内容将丢失
- **浏览器数据被清除**：用户清除了浏览器缓存或 localStorage，之前创建的旅行计划数据将永久丢失，系统应提示用户重新创建计划

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在旅行计划创建完成后，自动生成包含景点、住宿、餐饮、交通安排的完整行程规划
- **FR-002**: 系统必须提供旅行计划概览页，展示计划基本信息（目的地、日期范围、计划简介）
- **FR-003**: 系统必须在概览页以分类方式展示预算明细，包含景点门票、酒店住宿、餐饮交通、餐饮美食四个分类的预算金额和预估总费用
- **FR-004**: 系统必须在概览页展示目的地地图，并在地图上标注行程涉及的景点位置和住宿位置
- **FR-005**: 系统必须在概览页按天列出每日行程概览列表，每一天展示日期和行程概要
- **FR-006**: 用户必须可以通过点击概览页的某一天，进入该天的行程详情页
- **FR-007**: 系统必须提供逐日行程详情页，展示当日的日程安排（时间段与活动对应关系）
- **FR-008**: 系统必须在每日详情页以卡片形式展示景点详情，包含景点图片、名称、游玩时间、景点特色、推荐信息和详细描述
- **FR-009**: 系统必须在每日详情页展示住宿安排信息，包含酒店名称、房型、地址、入住时间和退房时间
- **FR-010**: 系统必须在每日详情页展示餐饮安排，包含早餐、午餐、晚餐的推荐方案
- **FR-011**: 系统必须在每日详情页展示交通安排信息，包含航班、火车等交通方式详情
- **FR-012**: 系统必须在每日详情页展示当日天气信息
- **FR-013**: 用户必须可以对某一天的行程进行编辑，包括调整景点顺序和删除景点
- **FR-014**: 用户编辑行程后必须可以保存，保存后页面展示更新后的行程内容
- **FR-015**: 用户必须可以将旅行计划导出为图片或 PDF 格式
- **FR-016**: 用户可以选择重新规划整个行程，系统必须在重新规划前展示确认提示
- **FR-017**: 用户确认重新规划后，系统必须重新生成旅行计划并更新页面内容
- **FR-018**: 如果行程生成、导出或保存失败，系统必须展示错误信息并提供重试选项
- **FR-019**: 系统必须支持用户在概览页和逐日详情页之间自由切换导航
- **FR-020**: 系统在加载行程数据时必须展示加载状态提示

### Key Entities

- **旅行计划概览 (Travel Plan Overview)**: 旅行计划的总览信息，包含目的地、日期范围、计划简介、预算分类明细、预估总费用
- **预算明细 (Budget Breakdown)**: 按分类组织的费用明细，包含景点门票、酒店住宿、餐饮交通、餐饮美食四个分类的金额
- **目的地地图 (Destination Map)**: 展示目的地地理信息的地图视图，标注行程中的景点位置和住宿位置
- **每日行程概览 (Daily Itinerary Summary)**: 某一天行程的概要信息，包含日期和行程简述，用于概览页展示
- **每日行程详情 (Daily Itinerary Detail)**: 某一天的完整行程信息，包含日程时间线、景点列表、住宿信息、餐饮安排、交通方案和天气
- **景点详情 (Attraction Detail)**: 某个景点的详细信息，包含图片、名称、游玩时间、景点特色、推荐信息、详细描述
- **住宿安排 (Accommodation Plan)**: 当日的住宿信息，包含酒店名称、房型、地址、入住退房时间
- **餐饮安排 (Dining Plan)**: 当日的餐饮推荐，包含早餐、午餐、晚餐的方案
- **交通安排 (Transportation Plan)**: 当日涉及的交通信息，包含航班、火车等
- **天气信息 (Weather Info)**: 当日目的地的天气情况

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户完成旅行计划创建后，概览页加载成功并在 3 秒内展示所有核心内容（预算、地图、每日行程列表）
- **SC-002**: 用户在概览页查看预算明细时，能在一眼看到分类预算和总费用，无需额外操作
- **SC-003**: 用户可以在 5 秒内从概览页导航到任意一天的行程详情页
- **SC-004**: 逐日行程详情页加载完成后，用户能在 10 秒内获取当天所有行程信息（景点、住宿、餐饮、交通、天气）
- **SC-005**: 用户编辑景点顺序或删除景点的操作在 3 步内完成，保存后更新立即生效
- **SC-006**: 用户导出旅行计划的操作在 10 秒内完成文件生成和下载
- **SC-007**: 90% 的用户可以在无需帮助的情况下顺利完成概览查看、逐日详情浏览、景点编辑和导出操作
- **SC-008**: 地图加载成功率和景点信息完整率达到 95% 以上

## Assumptions

- **A-001**: 本功能依赖 001 模块（旅行计划创建）生成的旅行计划数据，不包含独立的行程生成逻辑
- **A-002**: 地图展示使用高德地图 MCP 服务（API Key 通过环境变量 `AMAP_WEB_KEY` 配置），MCP 服务部署在后端作为代理层，调用高德 POI 搜索、天气查询、路线规划等 API，返回 JSON 格式数据用于景点和住宿位置标注
- **A-003**: 天气信息通过高德地图 MCP 服务的天气查询 API 获取，基于目的地城市和日期查询，行程生成时一次性获取并缓存
- **A-004**: 景点图片通过第三方图片服务（如项目环境变量 `UNSPLASH_ACCESS_KEY` 配置的 Unsplash）获取
- **A-005**: 导出的图片和 PDF 文件包含行程的核心信息（行程概览、预算、景点详情等），格式清晰可读
- **A-006**: 编辑行程操作仅影响当前旅行计划，不触发重新生成整个行程
- **A-007**: 重新规划操作会完全覆盖当前行程的所有编辑内容，用户无法恢复被覆盖的编辑
- **A-008**: 用户无需注册或登录即可使用旅行计划详情查看功能（延续 001 模块的假设），旅行计划数据仅存储在浏览器 localStorage 中，用户只能在同一设备同一浏览器上访问，关闭或清除浏览器数据后计划将丢失
- **A-009**: 用户拥有稳定的互联网连接以支持页面加载、地图渲染和图片获取
